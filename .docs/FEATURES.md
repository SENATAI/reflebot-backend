# FEATURES

## Работа с файлами Telegram

### Зафиксированное решение

Для Telegram workflow принято упрощённое решение по файлам:

- хранить только `telegram file_id`
- не строить связь между возвращаемым в Telegram файлом и внутренним файловым хранилищем
- при необходимости повторной отправки файла в Telegram возвращать именно `telegram_file_id`
- логику обратной выдачи файлов через отдельное хранилище, `presigned URL` и fallback-механизм отложить на потом

### Что это означает

На текущем этапе целевая логика должна быть такой:

- файл приходит в Telegram-бот
- бот получает от Telegram `file_id`
- backend сохраняет именно Telegram `file_id`
- при запросе на повторную отправку backend возвращает именно этот `file_id`
- дальше Telegram-бот сам использует этот `file_id` для повторной отправки пользователю через Telegram Bot API

### Что сознательно откладывается

Пока не реализуем:

- fallback на отдельное файловое хранилище при повторной отправке файла
- возврат файла пользователю через `presigned URL`
- смешанную модель `telegram_file_id + file record in storage`
- логику восстановления файла из внутреннего файлового хранилища, если Telegram `file_id` недоступен

### Статус

Это уже целевое направление текущей реализации reflections workflow.
